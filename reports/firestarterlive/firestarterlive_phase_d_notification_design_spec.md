# FirestarterLive Phase D Notification Design Specification

Date: 2026-06-10T12:21:55-04:00  
Status: **DESIGN_STAGE_ONLY**

---

## 1. Notification Adapter Priority Order
To maintain a secure, robust, and incremental rollout, notification adapters must be configured and activated in the following priority order:
1. **Console Adapter (Priority 1):** Logs notice messages directly to `stdout`/`stderr`. Always active.
2. **Local File Adapter (Priority 2):** Appends serialized event objects to `reports/firestarterlive/firestarterlive_alert_events.jsonl` (git-ignored).
3. **Slack Webhook Adapter (Priority 3):** Dispatches JSON payloads to a Slack channel via an incoming webhook URL. Optional and disabled by default.
4. **Other Adapters (Priority 4 - Future):** Discord, Telegram, Email, Windows Toast. Standardized interface but inactive/stubbed.

---

## 2. Rationale for Local-First (Console/File-Only) Priority
The first active adapters must be console and local-file only because:
- **Zero-Leak Security:** Eliminates the risk of leaking sensitive API keys, webhooks, or intellectual property over public networks.
- **Local Validation:** Allows verifying the stability and execution frequency of the watcher shell without network dependencies or API rate-limiting issues.
- **No Spam:** Prevents accidental notification fatigue during testing and local adjustments.

---

## 3. Fail-Closed Alert Settings Rules
The system must terminate execution with an exit code of `1` (fail-closed) if any of the following safety rules are violated:
- **Missing Secrets:** If any network adapter (e.g., Slack) is marked `enabled: true` under `firestarterlive_alert_settings.yaml`, but the corresponding environment variable (e.g., `FIRESTARTERLIVE_SLACK_WEBHOOK_URL`) is missing, empty, or invalid.
- **Safety Flags Violation:** If `safety.no_auto_trade` is false, `safety.no_order_execution` is false, or `safety.notice_only` is false.
- **Scenario Execution Allowed:** If any scenario has `trade_execution` set to true.

---

## 4. Payload Schema for Notice Delivery
Any alert event generated by the watcher must adhere to the following JSON schema:
```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "title": "FirestarterLiveNoticeEvent",
  "type": "OBJECT",
  "required": [
    "timestamp",
    "scenario_id",
    "severity",
    "message",
    "metrics_snapshot",
    "governance"
  ],
  "properties": {
    "timestamp": { "type": "STRING", "format": "date-time" },
    "scenario_id": { "type": "STRING" },
    "severity": { "type": "STRING", "enum": ["watch", "warning", "critical"] },
    "message": { "type": "STRING" },
    "metrics_snapshot": {
      "type": "OBJECT",
      "required": ["er", "fmlc", "flowprint"],
      "properties": {
        "er": { "type": ["number", "null"] },
        "fmlc": { "type": ["number", "null"] },
        "flowprint": { "type": ["number", "null"] },
        "raw_score": { "type": ["number", "null"] }
      }
    },
    "governance": {
      "type": "OBJECT",
      "required": [
        "NOTICE_ONLY",
        "RESEARCH_ALERT",
        "NO_AUTO_TRADE",
        "NO_EXCHANGE_KEYS",
        "NO_ORDER_EXECUTION",
        "MANUAL_REVIEW_REQUIRED"
      ],
      "properties": {
        "NOTICE_ONLY": { "type": "BOOLEAN", "const": true },
        "RESEARCH_ALERT": { "type": "BOOLEAN", "const": true },
        "NO_AUTO_TRADE": { "type": "BOOLEAN", "const": true },
        "NO_EXCHANGE_KEYS": { "type": "BOOLEAN", "const": true },
        "NO_ORDER_EXECUTION": { "type": "BOOLEAN", "const": true },
        "MANUAL_REVIEW_REQUIRED": { "type": "BOOLEAN", "const": true }
      }
    }
  }
}
```

---

## 5. Required Governance Fields
Every alert notification payload must carry the following metadata header block populated exactly as true:
- `NOTICE_ONLY`: Confirms this is purely an informational alert.
- `RESEARCH_ALERT`: States that the alert arises from statistical research rules only.
- `NO_AUTO_TRADE`: Declares that no trade was or will be automatically initiated.
- `NO_EXCHANGE_KEYS`: Declares that no exchange credentials or authentication keys are loaded.
- `NO_ORDER_EXECUTION`: Declares that no order routing features are compiled.
- `MANUAL_REVIEW_REQUIRED`: Asserts that manual chart review is required before taking any action.

---

## 6. Forbidden Fields
The notifier payload and scripts must **never** include or compute fields relating to live trade execution:
- `entry`: No entry price targets.
- `exit`: No exit target prices.
- `leverage`: No leverage multipliers (e.g. `20x`).
- `position size`: No target sizing in USD or contract counts.
- `order route`: No API paths or parameters for placing orders.
- `exchange key`: No storage or references to private API key strings.
- `profit claim`: No automated stop-loss or take-profit orders.

---

## 7. Secret-Handling Rule
- **No Hardcoded Keys:** Webhook endpoints, tokens, and Slack API credentials must never be written to configuration YAML/JSON files or python source scripts.
- **Environment Variable Isolation:** Webhook URLs must be loaded strictly at runtime using `os.environ.get("FIRESTARTERLIVE_SLACK_WEBHOOK_URL")` or corresponding variables.
- **Safety Verification:** The script must fail closed if the environment variable exists but is syntactically invalid (e.g., does not start with `https://hooks.slack.com/`).
